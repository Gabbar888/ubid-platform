A platform for Karnataka Commerce & Industries

Department of Commerce & Industries · Government of Karnataka

The problem

Karnataka's business-facing regulatory landscape is served by 40+ State department systems — Shop & Establishment, Factories, Labour, KSPCB, BESCOM, BWSSB, Fire, Food Safety…

Each was built in isolation

  • Own schema, own record IDs, own validation rules
  • Business name and address are free text
  • PAN and GSTIN only partially captured
  • No reliable join key across systems
  • Same business = different rows in different databases
Result: Karnataka C&I cannot today answer basic questions about its own industrial base — how many businesses are operating, in what sectors, where, and with what recent activity.

Two coupled sub-problems

Part A — Entity Resolution

Given master data from N State systems, automatically link records that refer to the same real-world business and assign each a single Unified Business Identifier (UBID).

Part B — Activity Inference

Given a stream of activity events (inspections, renewals, bills, returns), infer whether each UBID is currently Active / Dormant / Closed.

B is meaningless without A. Sequenced but mutually informing — a UBID never appearing in any event stream is suspicious; reviewer corrections to A change the evidence base for B.

Non-negotiables shaped the architecture

ConstraintArchitectural consequence
No source-system changesFederation layer; pull-based ingest only; never write back
Synthetic / scrambled data onlyNo memorising real entities; structural matching only
Decisions explainable + reversibleTwo-tier scoring + SHAP per pair; versioned linkage

ReasonDetail

Bonus: isotonic regression on top calibrates raw scores into probabilities. ECE = 0.013 — well-calibrated.

The 25-feature vector

Five groups, each captures a different signal type:

GroupFeatures
Name (5)Jaro-Winkler · token-set · Jaccard trigram · LCS ratio · exact-stripped
Address (6)pin eq · door eq · locality match · cross distance · Levenshtein · geo distance (km) ← Nominatim
Identifier (4)GSTIN eq · phone eq · email-domain eq · PAN agreement
Structural (4)sector NIC · legal-form · log employee-ratio · log reg-date diff
Blocking (6)shared PAN · derived PAN · pin+name · pin+door · phone · n_shared
Deliberately no neural embeddings. Bengaluru-specific business vocabulary (locality names, family-name compounds) is poorly represented in pretrained multilingual models. Probabilistic string distances give us decomposable, defensible features.

Two-tier scoring

Deterministic tier (auto-decides 30–50% of cases)

if PAN(a) == PAN(b) and pin(a) == pin(b):
    → match (probability 1.0)
if PAN(a) ≠ PAN(b) and both non-null:
    → non-match (probability 0.0)

Easy to explain to a reviewer ("these share PAN ABCDE1234F").

Probabilistic tier

For everything else: 25-feature vector → LightGBM → raw score → isotonic calibrator → calibrated probability.

Decision rule: p ≥ 0.95 → auto-link  ·  0.55 ≤ p < 0.95 → review queue  ·  p < 0.55 → reject

Inline clustering — single biggest improvement

  1. Block — find candidate pairs
  2. Score — LightGBM probability
  3. Union-find over auto-linked pairs (this batch + historical) respecting cannot-link constraints
  4. Each connected component = one UBID
  5. Multiple existing UBIDs merge if a new record connects them

Activity engine — cadence-aware decay

Each event type has a weight w and a cadence τ.

contribution(t) = w · exp( -Δt / α·τ )
Eventwτ (days)Sign
BESCOM bill paid0.930+
Factory Form 20 (annual)0.8365+
KSPCB CFO renewal0.9varies+
BESCOM zero consumption0.430−
S&E renewal pre-20190.71825+
Continuity score S = Σ contributions. Bands: S ≥ 1.5 → Active · 0.15 ≤ S < 1.5 → Dormant · else Closed-by-silence. Plus deterministic overrides for closure events. Sector priors widen the Dormant band for seasonal businesses.

Reviewer workflow

Review Queue — 4 decision types

  • Confirm match → must-link constraint + positive training label
  • Reject → cannot-link constraint + negative training label + split if currently merged
  • Defer to senior → priority boost
  • Flag quality → mark source record for data-quality review

Sorting Mat (Audit Merges)

For multi-record UBIDs (40+ records possible): sort each into Group A/B/C/Solo. One submit click → atomic regroup with hundreds of constraint writes.

Two-tier reviewers

Junior (5–10 min cases) · Senior (escalations, policy-setting precedents).

Active-learning loop — model gets better with every decision

  1. Reviewer decides → row written to training_labels
  2. POST /admin/retrain re-fits LightGBM + recalibrates isotonic
  3. A/B against previous model — pre/post Brier, ECE, F1 reported every run
  4. Retrain history chart (Admin page) shows model improving over time
  5. Constraint propagation — must-link / cannot-link survive future re-clusterings
  6. Smart re-score — only review-queue + boundary pairs after retrain (10 s vs hours at scale)
Every Sorting-Mat audit on a 30-record cluster generates ~600 labelled pairs in one click.

Self-hosted Nominatim — proposal-compliant geocoding

Why self-hosted rather than Google / HERE:

  • The proposal forbids hosted-API calls on PII addresses
  • Nominatim runs in our Docker stack, uses OpenStreetMap data
  • Free, on-prem, no rate limits, no data leaving the server

How it plugs in

  1. Adapter parses address → canonical record
  2. Geocoder asks Nominatim for (lat, lng)
  3. Falls back to a curated 85-entry locality dictionary if Nominatim misses
  4. Coordinates feed the addr_geo_distance_km feature in the scorer

Reversibility — every decision is undoable

Unmerge

Activity Status page — split one wrong record off into a new UBID + cannot-link.

Sorting Mat

Audit Merges — split N records into K sub-clusters in one transaction.

Reject

Review Queue — write cannot-link, split if currently merged.

Every operation logged in reviewer_decisions + linkage_constraints. The Audit Trail timeline on Activity Status shows the full history per UBID.

Quarantine — events never silently dropped

Activity events arrive keyed by source-system identifiers. Some can't be joined:

  • Source record not yet ingested
  • Linkage was below auto-link threshold
  • Sub-identifier not yet attached
These events go to a quarantine queue, indexed by source identifier, and re-played whenever the linkage table updates. Top-level dashboard metric. Never silently dropped.

The exemplar query — now answerable

"Active factories in pin code 560058 with no inspection in the last 18 months."

Today this is impossible in Karnataka. Factories live in FBIS, inspections in another DB, no join key.

curl -X POST http://localhost:8000/api/v1/query/active-businesses \
  -H "Content-Type: application/json" \
  -d '{
    "verdict": "active",
    "pin_code": "560058",
    "source_system": "fbis",
    "no_event_type": "fac_inspection",
    "no_event_since_days": 540
  }'

Returns a list of UBIDs with verdict, score, and supporting evidence — in milliseconds.

Risks & mitigations

RiskMitigation
BESCOM consumer name = landlord, not businessRisk flag + address-only matching with raised threshold
Wrong merge corrupts events for two businessesConservative auto-link, cannot-link constraints, reversibility
Reviewer bottleneckActive-learning queue ordering, bulk decisions
Calibration driftReliability diagrams refreshed weekly, retrain history chart
Schema drift in source systemsAdapters validate canonical schema, fail loudly
Pre-PAN-mandate recordsSoft-match with raised threshold, address-driven
Multi-state legal entityEmbedded PAN extracted from GSTIN; one Legal Entity, one UBID per state

What's still future work

  • Real Kafka wiring for activity events (currently synchronous HTTP)
  • Authentication (basic auth + role-based access)
  • Drift monitoring chart (calibration over time, feature distribution drift)
  • Time-travel view (any UBID, any past date)
  • Webhook notifications for verdict changes
  • More adapters (the architecture is designed for 40+ — we proved it with 5)
Project is at ~75% hackathon scope, 100% proposal compliance. Remaining items are production hardening, not feature gaps.
